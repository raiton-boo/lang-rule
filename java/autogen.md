# 自動生成コード（生成物）の取り扱い（詳細）

この文書は `java/java.md` の「自動生成コード」節の詳細補足です。OpenAPI, Protobuf, Thrift 等で生成されるコードの扱い方、名前付けポリシー、移行方針についてまとめます。

## 基本方針

- 生成物は原則として手で編集しない。生成ロジック（テンプレート・設定）を変更して対応する。
- プロジェクト内で生成コードの配置場所とパス名規約を決めておく（例: `generated/` ディレクトリ配下）。

## 命名の扱い

- 生成ツールに名前付けルールを適用できる場合は、プロジェクトの命名規則（UpperCamelCase / lowerCamelCase）に合わせる設定を行う。
- どうしても生成物に命名違反が残る場合は、変換レイヤー（ラッパー／アダプタ）で名前を整える。

## 変換レイヤーの例

- 生成 DTO をそのまま使わず、アプリケーション用の DTO に変換するファクトリ／マッパーを用意する（`GeneratedXxx -> DomainXxx`）。

## バージョン管理と自動生成

- 生成コードは可能なら CI の一部として生成プロセスを実行し、コミットするか、Artifact として管理する方針をチームで決める。
- 生成設定（OpenAPI spec / proto files）はソース管理し、生成手順を README に記載する。

## 移行方針

- 既存の手書きコードと生成コードが衝突する場合は、段階的に生成器の設定を調整し、互換レイヤーを用意して移行する。
